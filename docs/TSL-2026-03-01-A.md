# ACS TECHNICAL SESSION LOG
ID: TSL-2026-03-01-A
Módulo: Authentication Core / Session Engine / Login Hard Security
Estado: ABIERTO

---

## 1. Contexto General

Durante la evolución del sistema de autenticación de ACS se realizó la transición desde un login básico con validación de hash hacia un modelo de seguridad estructural con:

- Lock automático por intentos fallidos.
- Creación de sesión server-side.
- Validación de sesión obligatoria para endpoints protegidos.
- Integración de session_token en frontend.

Este TSL documenta la consolidación del Login Security Core y la implementación del Session Engine.

---

## 2. Problemas Detectados

### ❌ 2.1 Ausencia de Sesión Server-Side

El sistema retornaba:

{
  status: "HAS_AIRLINE",
  user: {...}
}

Pero no generaba session_token persistente en hoja SESSIONS.

Impacto:
- createAlert podía ejecutarse sin validación real.
- No existía expiración estructural.
- Seguridad solo basada en localStorage (no válida en producción real).

---

### ❌ 2.2 Bloque L-3 Ubicado Incorrectamente

El bloque de creación de sesión fue colocado dentro de:

if (data.action === "createAlert")

En lugar de:

if (data.action === "login")

Impacto:
- Login nunca generaba session_token.
- Arquitectura inconsistente.
- Confusión estructural en doPost.

---

### ❌ 2.3 Función validateSession Dentro de doPost

validateSession fue definida dentro del bloque doPost.

Impacto:
- Error: SyntaxError: Unexpected token 'catch'
- Desbalance estructural del try/catch.
- Apps Script no compilaba correctamente.

---

## 3. Acciones Correctivas Implementadas

### ✅ 3.1 Reubicación de L-3 (Create Server Session)

Se movió el bloque de creación de sesión al interior del login:

const sessionToken = Utilities.getUuid();
const createdAt = new Date();
const expiresAt = new Date(createdAt.getTime() + 2 * 60 * 60 * 1000);

sessionSheet.appendRow([
  sessionToken,
  userId,
  airline,
  createdAt,
  expiresAt,
  "",
  true
]);

Ahora login retorna:

{
  status: "HAS_AIRLINE",
  session_token: "...",
  user: {...}
}

Resultado:
- Sesión persistente en hoja SESSIONS.
- Expiración real (2 horas).
- Token único por login.

---

### ✅ 3.2 Separación Estructural de validateSession

validateSession fue movida fuera de doPost.

Estructura final:

function doPost(e) {
   try {
      ...
   } catch(err) {
      ...
   }
}

function validateSession(...) {
   ...
}

Resultado:
- Eliminado SyntaxError.
- Arquitectura limpia.
- Cumple separación lógica correcta.

---

### ✅ 3.3 Protección de Endpoint createAlert

createAlert ahora valida:

const sessionCheck = validateSession(ss, data.session_token);

Si no es válido:

status: "UNAUTHORIZED"

Resultado:
- Endpoints protegidos.
- No ejecución sin sesión activa.
- Seguridad coherente server-side.

---

### ✅ 3.4 Integración Frontend (login.html)

Se agregó almacenamiento del session_token en localStorage:

session_token: result.session_token

Ahora ACS_activeUser contiene:

{
  userId,
  airline_id,
  session_token,
  ...
}

Resultado:
- Token disponible para llamadas protegidas.
- Integración completa Backend ↔ Frontend.
